## 铁律（任何情况下不得违反）

### 铁律 1：先读后写
修改任何文件之前，必须先读完它。  
创建任何新功能之前，必须先阅读现有同类实现。

### 铁律 2：改前必提交
`git status` 显示有未提交改动时：**先 `git commit`，再做新改动**。

### 铁律 3：说完成前必须展示证据

以下声明，没有命令输出就等于没有完成：

| 声明 | 必须执行的验证命令 |
|------|--------------------|
| "已创建文件" | `ls -la 文件路径` 或 `Test-Path 路径` |
| "已删除文件" | 确认文件不存在 |
| "已修复 bug" | 运行复现步骤，展示修复后的输出 |
| "构建成功" | 展示 build 最后几行输出 |
| （根据项目补充更多） | |

**没有命令输出 = 没有完成。**

### 铁律 4：从不猜测路径
不确定文件在哪里，先 `find` / `ls`，不凭记忆写路径。

### 铁律 5：不信任自己的上一条输出
完成修改后，独立运行验证命令，不依赖推断。

## 标准工作序列

```
Step 1 — 理解：读相关文件，理解现有实现
Step 2 — 计划：用表格列出改哪些文件/改什么/为什么，等用户确认
Step 3 — 执行：按计划逐文件修改，不随意扩展范围
Step 4 — 验证：运行命令，将实际输出粘贴到回复中
Step 5 — 汇报：用表格总结，每项附上验证证据
```
