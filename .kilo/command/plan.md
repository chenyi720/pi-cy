## 执行步骤

**第一步：理解需求**

在写计划之前，先回答以下问题（用表格）：

| 问题 | 回答 |
|------|------|
| 用户想实现什么？ | |
| 当前代码/状态是什么？ | |
| 有哪些约束或风险？ | |
| 预计影响哪些文件？ | |

**第二步：制定计划**

每个任务粒度：2-5 分钟可完成，有明确的完成标准。

| # | 任务 | 目标文件 | 改动内容 | 验证方法 |
|---|------|----------|----------|----------|
| 1 | | | | |
| 2 | | | | |

**第三步：⛔ 停止，等待用户批准**

输出计划后，必须明确说：

> **请确认以上计划后，我再开始执行。**

在用户明确说"开始"/"执行"/"好的"之前，**禁止写任何代码**。

## 规则

- 计划粒度：每步 2-5 分钟，不能是"重构整个模块"这种大任务
- 每步必须有验证方法
- 不确定的地方必须在计划中标注"❓待确认"，不要自行决策
- 遵循 TDD、YAGNI、DRY 原则
